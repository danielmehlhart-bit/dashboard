Referee Summary

Finder reported 26 findings (124 pts). After challenger review: 4 critical confirmed, 2 critical downgraded to medium, 8 medium confirmed, 3 medium downgraded to low, 5 low confirmed, 4 low disproved. Adjusted score: 89 pts (17 findings confirmed/adjusted).

The platform's tenant isolation is structurally sound — RLS policies cover all tables with tenant_id checks. The biggest real risks are: signup hijack via metadata, client-side-only impersonation losing state on refresh, and missing indexes for scale. The "missing billing infrastructure" gap is real but expected at this stage.

🔴 Critical Findings

#1 — Signup Tenant Hijack via raw_user_meta_data

CRITICAL +10 ✅ CONFIRMED

handle_new_user() accepts tenant_id from client metadata. Anyone knowing a valid tenant UUID can register into that tenant as admin via the public Supabase Auth API.

Stell dir vor, jemand kennt die Hausnummer von Alex' Büro in der App. Er kann sich einfach selbst als Admin in Alex' Firma registrieren — ohne Einladung. Das ist wie wenn jeder, der deine Adresse kennt, sich selbst einen Hausschlüssel machen kann.
Referee: Confirmed. The Auth API is public — you can't protect raw_user_meta_data from client-side manipulation. Only invite-flow should assign to existing tenants. Self-service must always create new tenant.

Fix: In handle_new_user(), ignore tenant_id from metadata entirely. Only accept it from a server-side invite function that sets it via admin.updateUserById.

#2 — Client-Side Impersonation Loses State on Refresh

CRITICAL +10 ⚠️ DOWNGRADED → MEDIUM +5

TenantContext is pure React state (useState). Refresh → sudo sees ALL tenants' data blended together.

Wenn du als Plattform-Admin (sudo) in Alex' Firma "reingehst" und dann die Seite neu lädst, vergisst die App, dass du gerade bei Alex bist. Du siehst dann plötzlich Daten von ALLEN Firmen durcheinander. Wie wenn du in einem Bürogebäude den Aufzug nimmst, in Alex' Etage bist, und nach einem Stromausfall plötzlich alle Türen offen stehen.
Challenger: While true that state is lost, sudo's RLS policies are read-only cross-tenant. Sudo can't accidentally write to wrong tenant because INSERT policies use auto_set_tenant_id (always sets to sudo's own tenant). The risk is data display leakage post-refresh, not data corruption.
Referee: Downgrade to Medium. Still needs fixing (sessionStorage minimum), but the blast radius is limited to read-only cross-tenant display for a single privileged user, not a security breach.

#3 — Cross-Tenant Notification Injection

CRITICAL +10 ⚠️ DOWNGRADED → MEDIUM +5

Notification INSERT policy checks is_team_member() but not that user_id target belongs to same tenant.

Theoretisch könnte jemand aus Firma A eine Benachrichtigung an jemanden in Firma B schicken. In der Praxis kommt die Nachricht aber nie an, weil der Empfänger sie nie sehen kann (sein Filter zeigt nur seine eigene Firma). Es ist wie ein Brief an die falsche Adresse — er kommt an, aber niemand macht ihn auf.
Challenger: The auto_set_tenant_id trigger sets tenant_id to inserter's tenant. The notification SELECT policy is user_id = auth.uid() AND tenant_id = get_user_tenant_id(auth.uid()). So even if Tenant A inserts a notification for a Tenant B user, that user will never see it because their SELECT policy filters on their own tenant_id. The notification exists in DB but is invisible to the target.
Referee: Downgrade to Medium. It's a data hygiene issue (orphaned rows) not a security breach. The notification is effectively dead on arrival. Still worth adding a WITH CHECK on user_id belonging to same tenant.

#4 — Storage Bucket Missing Tenant RLS

CRITICAL +10 ✅ CONFIRMED

No storage policies found in any migration for project-files or project-images buckets.

Alle hochgeladenen Dateien (Pläne, Fotos, Dokumente) liegen in einem Speicher ohne Zugangsregeln pro Firma. Wenn jemand den Dateipfad kennt oder errät, kann er die Dateien einer anderen Firma runterladen. Wie ein Aktenschrank ohne Schloss — wer die Schublade findet, kann reinschauen.
Referee: Confirmed. Storage RLS is separate from table RLS. If buckets are set to "authenticated" (Supabase default), any logged-in user from any tenant can read/write files if they know the path pattern. Needs immediate attention.

Fix: Add storage policies using path-based tenant isolation: storage.objects policies that check project ownership via is_project_accessible().

#5 — useTeamMembers() Leaks All Profiles for Sudo

CRITICAL +10 ✅ CONFIRMED

No impersonatedTenantId filter applied. Sudo sees all users across all tenants in team view.

Wenn du als Plattform-Admin die Team-Seite einer bestimmten Firma öffnest, siehst du nicht nur deren Mitarbeiter, sondern ALLE User von ALLEN Firmen — inkl. Namen und E-Mail-Adressen. Wie wenn du in Alex' Büro gehst und versehentlich die Personalakten aller Büros auf dem Tisch liegen.
Referee: Confirmed. useTeamMembers() has zero tenant filtering. Combined with "Sudo can read all profiles" RLS, this returns the entire user base. PII exposure (emails, names) of all tenants when sudo navigates to team page.

#6 — auto_set_tenant_id Bypass via Explicit tenant_id

CRITICAL +10 ✅ CONFIRMED

Trigger only fires IF NEW.tenant_id IS NULL. Explicit cross-tenant tenant_id on INSERT bypasses it.

Es gibt einen Automatismus, der bei neuen Einträgen (Projekte, Aufgaben etc.) automatisch die richtige Firma zuordnet. Aber: Wenn jemand absichtlich die Firma-ID einer anderen Firma mitschickt, wird der Automatismus übersprungen. Bei manchen Tabellen fängt die Datenbank das ab, bei ein paar aber nicht. Wie ein Sicherheitsschloss, das nur greift wenn man keinen eigenen Schlüssel mitbringt.
Challenger: Attempted disprove — most INSERT policies include tenant_id = get_user_tenant_id(auth.uid()) in WITH CHECK, which would block cross-tenant inserts. However, several exceptions exist:
• "Team can insert projects" → WITH CHECK only checks is_team_member(), no tenant_id validation
• forecast_cache_admin_insert → WITH CHECK only checks has_role(admin)
Referee: Confirmed for those specific tables. The trigger should always override: remove the NULL guard.

🟠 Medium Findings

#7 — Missing Indexes on tenant_id Columns

MEDIUM +5 ✅ CONFIRMED
Referee: No indexes on any tenant_id column across 10+ tables. Confirmed via migration review. Fine at 1-3 tenants, will degrade at 50+.
Jede Abfrage muss aktuell ALLE Daten durchsuchen, um die richtige Firma zu finden — wie ein Telefonbuch ohne Alphabet-Sortierung. Bei 3 Firmen egal, bei 50+ wird die App spürbar langsamer. Fix: Indexe anlegen (ein SQL-Befehl pro Tabelle, dauert 15 Min).

#8 — No Pagination in Admin Tables

MEDIUM +5 ✅ CONFIRMED
Referee: Both TenantList and AllUsersTable load everything. OK for now (<5 tenants), must fix before 50+.
Die Admin-Tabellen laden alle Firmen und alle User auf einmal in den Browser. Bei 500 Firmen mit je 10 Usern wären das 5.000 Datensätze auf einen Schlag. Fix: Seitenweise laden (Pagination), wie bei Google-Suchergebnissen.

#9 — get_user_tenant_id() Per-Row in RLS

MEDIUM +5 ⚠️ DOWNGRADED → LOW +1
Challenger: PostgreSQL evaluates STABLE functions once per transaction, not per row. With the function being LANGUAGE sql STABLE, the planner can inline and cache the result. The JWT-claim approach is better long-term but this isn't a current bottleneck.
Referee: Downgrade to Low. Good optimization to do eventually, but not a scalability blocker at current scale.

#10 — No Sudo Audit Trail

MEDIUM +5 ✅ CONFIRMED
Referee: Confirmed. No logging of sudo tenant entry/exit. Needed for accountability and GDPR compliance.
Wenn du als Plattform-Admin in eine Firma reinschaust, wird das nirgends protokolliert. Kein Log, wann du rein bist, was du gesehen hast, wann du raus bist. Für DSGVO und Vertrauen bei Kunden brauchst du das — wie ein Besucherbuch am Empfang.

#11 — Inconsistent Impersonation Filtering

MEDIUM +5 ✅ CONFIRMED
Referee: Only useProjects and useTasks (partially) apply impersonation filter. 6+ other hooks (documents, time entries, team members, project updates, notifications) do not. Confirmed via grep: only 8 occurrences of impersonatedTenantId in 800+ line file with 49 exported hooks.
Wenn du als Admin bei Alex' Firma "reingehst", zeigen nur Projekte und Aufgaben Alex' Daten. Dokumente, Zeiteinträge, Team-Ansicht, Updates, Benachrichtigungen — die zeigen noch alles durcheinander. Es ist als ob nur 2 von 10 Zimmern im Büro den richtigen Türschild haben.

#12 — project_members Missing tenant_id

MEDIUM +5 ⚠️ DOWNGRADED → LOW +1
Challenger: project_members RLS uses is_project_accessible() which already checks project.tenant_id. Isolation is maintained via the project's tenant ownership. Adding tenant_id is redundant for security, only useful for direct cleanup queries.
Referee: Downgrade to Low. Security is covered; it's a convenience gap for admin operations.

#13 — No Billing / Tenant Status Model

MEDIUM +5 ✅ CONFIRMED
Referee: Confirmed. The tenants table has only id/name/slug/created_at. No status, plan, seats_limit, or billing_customer_id. Expected at current stage but P0 before any paid tenant goes live.
Es gibt keine Möglichkeit, eine Firma zu pausieren, zu sperren oder deren Abo-Status zu tracken. Wenn Alex nicht zahlt, kannst du ihn nicht aussperren ohne direkt in die Datenbank zu gehen. Für den Piloten OK, aber bevor du Geld verlangst, brauchst du mindestens ein Status-Feld (aktiv/gesperrt/trial).

#14 — Admin Role Before Email Verification

MEDIUM +5 ⚠️ DOWNGRADED → LOW +1
Challenger: Supabase has email confirmation enabled by default. handle_new_user() fires on INSERT ON auth.users which happens at signup, but the user can't get a valid session until email is confirmed (if enabled). The role exists in DB but is unusable without a session.
Referee: Downgrade to Low. Depends on Supabase project auth settings. If email confirmation is enabled (default), this is not exploitable. Worth verifying the setting.

#15 — Raw DB Errors in CreateTenantDialog

MEDIUM +5 ✅ CONFIRMED
Referee: Confirmed. toast.error("Fehler beim Anlegen: " + error.message) passes raw Postgres error to UI. Minor info disclosure + bad UX.
Wenn beim Anlegen einer Firma was schiefgeht, sieht der User eine kryptische Datenbank-Fehlermeldung statt "Dieser Name ist schon vergeben". Peinlich, aber nicht gefährlich.

#16 — 3 DB Roundtrips for Task Update

MEDIUM +5 ✅ CONFIRMED
Referee: Confirmed. SELECT old → UPDATE → SELECT new. Move to Postgres trigger for notifications.
Wenn jemand eine Aufgabe ändert, macht die App 3 einzelne Datenbank-Anfragen hintereinander statt einer. Bei 50 Leuten gleichzeitig wird das langsam. Lösung: Die Datenbank soll die Benachrichtigung automatisch auslösen (Trigger), statt dass die App es per Hand macht.
