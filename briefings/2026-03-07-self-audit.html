🚨 Heutige Fehler — Konkretes Beispiel kritisch

1. Falsche Fehlerdiagnose: gog vs. Timeout

Was passiert ist: Ich habe "gog braucht Keyring-Passwort" gemeldet. Der echte Fehler war ein Timeout (120s). gog ist seit Wochen durch gws ersetzt — ich hätte das wissen müssen.
Root Cause: Ich habe die tatsächliche Cron-Fehlermeldung nicht gelesen, sondern eine plausible Erklärung halluziniert. Sauber wäre gewesen: lastError lesen, dann antworten.

2. Briefings auf falschem Board

Was passiert ist: Alle Cron-Telegram-Pings verlinkten auf danielmehlhart-bit.github.io/dashboard/ statt mc.mehlhart.de. Mission Control wurde am 05.-06. März eingerichtet — ich hätte die Crons direkt damals updaten müssen.
Root Cause: Es gibt keinen Mechanismus, der "neue Entscheidung in MEMORY.md" → "propagiere in betroffene Crons" bridged. Jede Entscheidung ist ein Silo.

3. Sub-Agent modifiziert AGENTS.md

Was passiert ist: Ein Sub-Agent (mc-deploy Workflow) hat AGENTS.md eigenständig editiert. Core-Files sind Read-Only-Policy — diese Guardrail existiert zwar in AGENTS.md, aber Sub-Agents ignorieren sie.
Root Cause: Sub-Agent-Prompts haben keinen expliziten "Du darfst AGENTS.md nicht editieren"-Hinweis. Nur implizite Policy.

4. OAuth-Chaos bei Gmail Draft

Was passiert ist: 5 Iterationen für einen Gmail Draft: falscher Scope → falsche Client-Type → 400 → 400 → endlich Desktop-App.
Was gut war: Ich habe das Problem strukturiert debuggt. Ergebnis: gws hat jetzt Write-Scope. Einmaliger Aufwand mit dauerhaftem Nutzen.

🏗️ Architektur-Vorschläge

A — MEMORY.md in 3 spezifische Dateien aufspalten

Statt einer 240-Zeilen-Datei die alles enthält:

  • SETUP.md — Pi, Bot, Kanäle, API-Keys, Credentials. Selten geändert, immer korrekt.
  • PROJECTS.md — Aktive Projekte, Status, Repos, Ansprechpartner. Wöchentlich geupdated.
  • RULES.md — Operationale Regeln (Email-Policy, Telegram-Format, Sub-Agent-Regeln). Unveränderliche Policies.

MEMORY.md bleibt als persönliche Langzeit-Erinnerung (Daniels Charakter, Präferenzen) — aber nicht mehr als Operations-Manual.

B — Cron-Config-Check als Standardschritt

Jedes Mal wenn ich eine systemweite Entscheidung treffe (neue URL, neues Format, neues Tool), direkt danach: "Welche Crons sind davon betroffen?" — und diese updaten. Nicht als optionaler Schritt, sondern als Pflicht-Abschluss jeder Systemänderung.

Konkret: Neues Protokoll nach Systemänderung → exec cron list → grep nach betroffenen Strings → update.

C — Daily Memory in 2-Layer-Struktur

Statt N Dateien pro Tag:

  • memory/YYYY-MM-DD.md — Einzige Tages-Datei. Raw Log. Kein Threshold.
  • memory/changelog.md — Nur Decisions, Systemänderungen, neue Erkenntnisse. Maschinenlesbar (Date | Category | Change).

changelog.md ersetzt decisions.md + lessons-learned.md + den Decisions-Block aus MEMORY.md.

D — Workspace-Root Cleanup + Archiv

  • Ordner workspace/archive/YYYY-MM/ anlegen
  • Alle Dateien älter als 30 Tage aus Root → archive/
  • Monatliche Cleanup-Routine (z.B. 1. des Monats via Cron)
  • Nur Live-Dateien im Root: *.md Configs, aktive Skills

E — Sub-Agent-Guardrail hardcoden

In jeden Sub-Agent-Prompt explizit rein: "Du darfst NIEMALS folgende Dateien editieren: AGENTS.md, SOUL.md, MEMORY.md, SECURITY.md, USER.md. Nur Daniel darf diese Dateien ändern."

Skill-Templates (coding-agent, spawn-and-deliver) als erste Stelle — dort einbauen, dann automatisch in alle künftigen Spawns.

F — Diagnostik-First-Regel (für mich)

Wenn ein System Fehler meldet: Erst den tatsächlichen Fehlerstatus lesen, dann Diagnose stellen. Nie eine Fehlermeldung erfinden weil sie "wahrscheinlich klingt".

Konkret: cron runs <jobId> → letzten Run-Output lesen → dann erst antworten.
