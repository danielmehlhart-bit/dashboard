CRITICAL

C-1 · Hardcoded Encryption Key

CONFIRMED

📁 Datei

supabase/functions/_shared/encryption.ts

🔍 Problem

Die Funktionen encrypt() und decrypt() verwenden den Literal-String 'your-super-secret-key' als pgp_sym_encrypt-Key. Dieser Placeholder wurde nie durch einen echten Secret ersetzt — obwohl in SECURITY_FIXES_REALLY_APPLIED.md ein "C-01 Fix" dokumentiert ist.

-- Aktueller Zustand (VERWUNDBAR)
"SELECT pgp_sym_encrypt($1, 'your-super-secret-key')", [value]
"SELECT pgp_sym_decrypt(decode($1, 'hex'), 'your-super-secret-key')", [encryptedHex]

💥 Impact

Jeder mit Code-Zugriff (public repo, Insider, Leak) kennt den Key und kann alle gespeicherten Google OAuth access_tokens und refresh_tokens aller Tenants entschlüsseln. Direkt aus dem Supabase Dashboard. Vollzugriff auf alle verknüpften Gmail-Postfächer. Die Verschlüsselung erfüllt ihren Zweck nicht.

✅ Konkreter Fix

1. Neuen zufälligen 32-Byte-Key generieren und als Supabase Secret setzen:
2. Code auf Deno.env.get('ENCRYPTION_KEY') umstellen:
3. Alle bereits gespeicherten Tokens neu verschlüsseln (Migration nötig).

# Schritt 1: Sicheren Key generieren
openssl rand -hex 32

# Schritt 2: Als Supabase Secret setzen
supabase secrets set ENCRYPTION_KEY=<generated_key>

# Schritt 3: Code-Änderung
const key = Deno.env.get('ENCRYPTION_KEY');
if (!key) throw new Error('ENCRYPTION_KEY not set');
await client.query(
  "SELECT pgp_sym_encrypt($1, $2)", [value, key]
);

C-2 · oauth_state Tabelle ohne Row Level Security

CONFIRMED

📁 Datei

supabase/migrations/20260305224317_create_oauth_state_table.sql

🔍 Problem

Die oauth_state Tabelle wurde ohne ENABLE ROW LEVEL SECURITY und ohne eine einzige Policy erstellt. Jeder authentifizierte User kann States aller anderen User lesen, löschen oder mit fremder user_id einfügen.

💥 Impact

Token-Hijacking: State-Token eines anderen Users stehlen → für eigenen OAuth-Callback nutzen → OAuth-Token bindet sich an falschen Account. OAuth-Flow-Sabotage für andere User durch State-Deletion.

✅ Konkreter Fix

-- Migration hinzufügen
ALTER TABLE public.oauth_state ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can manage own oauth state"
  ON public.oauth_state FOR ALL
  TO authenticated
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

-- Service-Role für Callbacks benötigt separaten Ansatz:
-- oauth-callback Function nutzt Service Role → kein Problem
-- ABER: Insert in oauth-init muss user_id = auth.uid() erzwingen

C-3 · create-gedanke Edge Function ohne Authentication

CONFIRMED

📁 Datei

supabase/functions/create-gedanke/index.ts

🔍 Problem

Kein Bearer-Token-Check, kein auth.getUser(). Die userId kommt direkt aus dem Request-Body. Service-Role wird für alle DB-Operationen verwendet.

// VERWUNDBAR
const { transcript, userId, projectHint } = await req.json();
// KEIN authHeader check, KEIN getUser()!
const supabase = createClient(SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY);
await supabase.from("private_notes").insert({ user_id: userId, ... })

💥 Impact

Anonymer Angreifer kann private Notizen für beliebige User-UUIDs erstellen. Außerdem: API-Quota Exhaustion durch unauthentifizierte Massenrequests.

✅ Konkreter Fix

// Auth-Check am Anfang der Handler-Funktion
const authHeader = req.headers.get('Authorization');
if (!authHeader?.startsWith('Bearer ')) {
  return new Response('Unauthorized', { status: 401 });
}

const userClient = createClient(SUPABASE_URL, SUPABASE_ANON_KEY, {
  global: { headers: { Authorization: authHeader } }
});
const { data: { user }, error } = await userClient.auth.getUser();
if (error || !user) {
  return new Response('Unauthorized', { status: 401 });
}

// userId aus Token, NICHT aus Request-Body
const userId = user.id;

C-5 · delete-team-member ohne Cross-Tenant-Check

CONFIRMED

📁 Datei

supabase/functions/delete-team-member/index.ts

🔍 Problem

Die Funktion prüft ob der Caller Admin ist — prüft aber nicht, ob die Ziel-user_id zum selben Tenant gehört. adminClient.auth.admin.deleteUser() bypassed alle RLS.

const { user_id } = body; // UNKONTROLLIERT aus Request-Body

💥 Impact

Admin von Tenant A kann Admin von Tenant B (oder jeden beliebigen User) permanent löschen. Cascade-Deletes. DoS gegen andere Tenants.

✅ Konkreter Fix

// Tenant-Check VOR deleteUser
const callerTenantId = callerProfile.tenant_id;

const { data: targetProfile } = await adminClient
  .from('profiles')
  .select('tenant_id')
  .eq('id', user_id)
  .single();

if (!targetProfile || targetProfile.tenant_id !== callerTenantId) {
  return new Response('Forbidden: cross-tenant delete', { status: 403 });
}

// Erst jetzt löschen
await adminClient.auth.admin.deleteUser(user_id);

C-6 · invite-team-member — Sudo Privilege Escalation

CONFIRMED

📁 Datei

supabase/functions/invite-team-member/index.ts

🔍 Problem

Das role-Feld wird unvalidiert aus dem Request-Body übernommen und direkt in raw_app_meta_data geschrieben. Der handle_new_user()-Trigger liest diese Rolle und setzt sie dauerhaft.

const { email, role, first_name, last_name, project_ids } = body;
// KEINE Prüfung ob role != 'sudo'!
await adminClient.auth.admin.inviteUserByEmail(email, {
  data: { role: role, tenant_id: callerTenantId },
});

💥 Impact

Vollständige Platform Kompromittierung: Ein Admin (nicht sudo) kann { role: "sudo" } in den Invite setzen. Nach Anmeldung hat der neue User Zugriff auf /admin, alle Tenants, alle Finanzdaten, alle User-Löschfunktionen.

✅ Konkreter Fix

// Erlaubte Rollen für Admin-Caller
const ALLOWED_ROLES_FOR_ADMIN = ['member', 'lead', 'architect', 'client'];
const ALLOWED_ROLES_FOR_SUDO  = [...ALLOWED_ROLES_FOR_ADMIN, 'admin'];

const callerRole = callerProfile.role;

if (callerRole === 'sudo') {
  if (!ALLOWED_ROLES_FOR_SUDO.includes(role)) {
    return new Response('Invalid role', { status: 400 });
  }
} else if (callerRole === 'admin') {
  if (!ALLOWED_ROLES_FOR_ADMIN.includes(role)) {
    return new Response('Forbidden: only sudo can assign admin/sudo', { status: 403 });
  }
} else {
  return new Response('Forbidden', { status: 403 });
}

C-7 · task_comments — Alte INSERT-Policy nie gelöscht (Cross-Tenant Injection)

CONFIRMED

📁 Dateien

20260215160000_task_comments_notifications.sql + 20260216161224_35f09525.sql

🔍 Problem

Die alte Policy "Users can insert own comments" (WITH CHECK (author_id = auth.uid())) wurde nie gedroppt. PostgreSQL OR-verknüpft permissive Policies → reicht eine erfüllte Policy. Jeder authentifizierte User aus jedem Tenant kann Kommentare auf Tasks anderer Tenants inserieren.

💥 Impact

Cross-Tenant Kommentar-Spam, XSS-Payloads im JSONB-Content, Social Engineering gegen andere Tenants. UPDATE-Policy ohne Tenant-Check ermöglicht nachträgliche Manipulation.

✅ Konkreter Fix

-- Migration zum Droppen veralteter Policies
DROP POLICY IF EXISTS "Users can insert own comments" ON public.task_comments;
DROP POLICY IF EXISTS "Users can update own comments" ON public.task_comments;
DROP POLICY IF EXISTS "Users can delete own comments" ON public.task_comments;
DROP POLICY IF EXISTS "Project members can view comments" ON public.task_comments;

-- Saubere Ersatz-Policy (falls noch nicht vorhanden)
CREATE POLICY "Comments: only accessible project members"
  ON public.task_comments FOR ALL
  TO authenticated
  USING (
    is_project_accessible((
      SELECT project_id FROM public.tasks WHERE id = task_id
    ))
  )
  WITH CHECK (
    author_id = auth.uid() AND
    is_project_accessible((
      SELECT project_id FROM public.tasks WHERE id = task_id
    ))
  );

C-8 · project_members Policy — Cross-Tenant Membership möglich

CONFIRMED

📁 Datei

supabase/migrations/20260216161224_35f09525-f32a-40ec-ae69-c758dda4b736.sql

🔍 Problem

Die Policy "Admins can manage all project members" prüft ob der Caller Admin ist — aber nicht ob das Ziel-project_id zum selben Tenant gehört.

CREATE POLICY "Admins can manage all project members"
  ON public.project_members FOR ALL
  USING (public.has_role(auth.uid(), 'admin'));
  -- FEHLT: WITH CHECK (tenant_id = get_user_tenant_id(auth.uid()))

💥 Impact

Admin von Tenant A kann sich (oder einen User) in Projekte von Tenant B eintragen → is_project_accessible() gibt danach TRUE zurück → Vollzugriff auf alle Daten des Tenant-B-Projekts (Tasks, Docs, Finanzen, Bautagebuch).

✅ Konkreter Fix

-- Policy ersetzen
DROP POLICY "Admins can manage all project members" ON public.project_members;

CREATE POLICY "Admins can manage project members in own tenant"
  ON public.project_members FOR ALL
  TO authenticated
  USING (
    public.has_role(auth.uid(), 'admin') AND
    (
      SELECT tenant_id FROM public.projects WHERE id = project_id
    ) = public.get_user_tenant_id(auth.uid())
  )
  WITH CHECK (
    public.has_role(auth.uid(), 'admin') AND
    (
      SELECT tenant_id FROM public.projects WHERE id = project_id
    ) = public.get_user_tenant_id(auth.uid())
  );
